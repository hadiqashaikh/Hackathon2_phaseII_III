# API Contracts: Task Management Backend

## Overview
This document defines the API contracts for the task management backend, ensuring consistency in authentication, data formats, and error handling. The API is designed to work with JWT tokens generated by Better Auth and enforce strict user data isolation.

## Base URL
```
http://localhost:8000 (development)
```

## Authentication
All task management endpoints require JWT authentication via the Authorization header:

```
Authorization: Bearer <jwt_token>
```

## Common Response Formats

### Success Response
```json
{
  "id": "uuid-string",
  "title": "string",
  "description": "string or null",
  "completed": "boolean",
  "user_id": "string",
  "created_at": "ISO 8601 datetime",
  "updated_at": "ISO 8601 datetime"
}
```

### Error Response
```json
{
  "detail": "error message"
}
```

## API Endpoints

### Create Task
```
POST /api/tasks/
```

**Headers**:
```
Authorization: Bearer <jwt_token>
Content-Type: application/json
```

**Request Body**:
```json
{
  "title": "string (required, 1-255 chars)",
  "description": "string or null (max 1000 chars)",
  "completed": "boolean (default: false)"
}
```

**Response (Success)**:
- Status: 200 OK
```json
{
  "id": "uuid-string",
  "title": "string",
  "description": "string or null",
  "completed": "boolean",
  "user_id": "string (extracted from JWT)",
  "created_at": "ISO 8601 datetime",
  "updated_at": "ISO 8601 datetime"
}
```

**Response (Error)**:
- Status: 401 Unauthorized
```json
{
  "detail": "Could not validate credentials"
}
```
- Status: 422 Unprocessable Entity
```json
{
  "detail": "Validation error details"
}
```

## JWT Token Validation Contract

### Expected Token Structure
The backend expects JWT tokens in this structure:
```json
{
  "id": "user_id_string",
  "email": "user_email",
  "name": "user_name",
  "iat": 1234567890,
  "exp": 1234571490
}
```

### Validation Requirements
1. Token must be properly signed with the BETTER_AUTH_SECRET
2. Token must not be expired (check exp claim)
3. Token must contain a valid user "id" field
4. Tokens must use HS256 algorithm

### Error Handling
- Invalid signature: 401 Unauthorized
- Expired token: 401 Unauthorized
- Missing user ID: 401 Unauthorized
- Malformed token: 401 Unauthorized

## Multi-tenancy Contract

### Data Isolation Rules
1. All queries must filter by user_id from JWT token
2. Users cannot access tasks belonging to other users
3. Task creation automatically assigns user_id from JWT token
4. URL parameters must match authenticated user identity where applicable

### Error Scenarios
- Attempting to access another user's task: 404 Not Found
- Attempting to update another user's task: 404 Not Found
- Attempting to delete another user's task: 404 Not Found

## Security Requirements

### Input Validation
- Title: 1-255 characters
- Description: Max 1000 characters (optional)
- Completed: Boolean value only

### Rate Limiting
- TBD (may implement per user/IP basis)

### Logging Requirements
- Authentication failures should be logged for security analysis
- Successful operations should be logged for audit purposes
- No sensitive data (like passwords) should be logged

## CORS Policy
- Allow requests from frontend domain where the Better Auth is implemented
- Only allow credentials to be sent with requests from trusted origins
